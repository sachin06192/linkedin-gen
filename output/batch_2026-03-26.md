### Post 4: Listicle / Tips
IMAGE_QUERY: quantization neural network model compression diagram explanation

5 things I learned from ngrok's "Quantization from the Ground Up" post this week:

**1. Quantization isn't just "make the numbers smaller."** It's choosing which information to throw away. Think of it like JPEG compression for neural networks — you're trading precision for speed, and the art is knowing where precision matters.

**2. INT8 quantization cuts model size by 4x with ~1% accuracy loss.** For most production use cases, that's a no-brainer. If you're serving an LLM at full FP32 precision, you're burning money.

**3. The calibration dataset matters more than the technique.** I ran the same quantization method with two different calibration sets. One gave me 2% accuracy loss. The other gave me 12%. The algorithm was identical — the data wasn't.

**4. Post-training quantization is free lunch, but QAT is the real meal.** Quantization-Aware Training bakes the precision loss into the training process itself. It takes longer but the results are dramatically better for aggressive quantization (INT4 and below).

**5. You can quantize selectively.** Attention layers are precision-sensitive. Feed-forward layers are not. Quantize the insensitive layers aggressively and leave the critical ones alone. This hybrid approach gets you 80% of the size savings with 10% of the accuracy cost.

If you're deploying any ML model to production, understanding quantization isn't optional anymore. It's the difference between a $500/month GPU bill and a $50/month one.

### Post 8: Edutainment: 1000 Hours -> 5 Minutes
IMAGE_QUERY: model quantization INT8 FP16 comparison chart performance

I spent 3 weeks understanding LLM quantization. Here's what actually matters.

**The one-sentence version:** Quantization shrinks a model by using fewer bits per number. A 70B parameter model at FP16 (16 bits) takes ~140GB. At INT4 (4 bits) it takes ~35GB. Same model, fits on your laptop.

**Why it works:** Neural networks are surprisingly tolerant of imprecision. The difference between a weight of 0.23847 and 0.24 doesn't matter for most predictions. Quantization exploits this.

**The 3 methods you actually need to know:**

1. **Post-Training Quantization (PTQ):** Take a trained model, shrink it. Fast, free, works for INT8. Think of it as "quick and dirty."

2. **GPTQ:** A smarter PTQ that analyzes which weights are most important and preserves those with higher precision. This is what most Hugging Face quantized models use.

3. **GGUF (llama.cpp):** Quantization format specifically for running models locally. If you've ever downloaded a model from TheBloke on Hugging Face, it was probably GGUF.

**The cheat sheet:**
- INT8 → almost no quality loss, 2x smaller
- INT4 (Q4_K_M) → noticeable but usable, 4x smaller
- INT2 → significant quality loss, only for experiments

**When NOT to quantize:** Anything doing math, code generation with precise syntax, or tasks where subtle wording matters. For summarization, classification, and chat? Quantize aggressively.

Start with Q4_K_M in GGUF format. It's the sweet spot for 90% of use cases.

### Post 11: Edutainment: How It Actually Works
IMAGE_QUERY: ARC-AGI benchmark test AI reasoning visual puzzles grid examples

ARC-AGI-3 just dropped. Everyone's talking about it. Almost nobody understands what it actually tests.

Here's how it works, simply.

ARC = Abstraction and Reasoning Corpus. It's a set of visual puzzles — grids of colored cells. Each puzzle gives you a few input/output examples, and you have to figure out the transformation rule, then apply it to a new input.

Think of it like an IQ test for AI. Not "memorize Wikipedia." Not "generate fluent text." Can you look at 3 examples and *infer the pattern*?

**Example:** Three examples show a 3x3 grid where the blue cells rotate 90 degrees clockwise. The test input is a new grid. Can you rotate correctly?

Humans solve these at ~85% accuracy in a few seconds. GPT-4 gets ~5%. Claude gets ~21%. The best AI system (with specialized architectures) got ~55% on ARC-AGI-2.

**Why this matters:**

Current LLMs are trained on text patterns. They're amazing at tasks that look like their training data. ARC tests something fundamentally different: few-shot *visual* reasoning from first principles.

**ARC-AGI-3 makes it harder:**
- More abstract rules
- Fewer examples per puzzle
- Rules that require combining multiple transformations

The $1M prize is still unclaimed. The benchmark exists specifically because it tests the gap between "pattern matching at scale" and "actual reasoning."

If you want to understand where AI genuinely is vs. where the hype says it is, ARC-AGI is the scoreboard that matters.

### Post 17: Listicle / Tips

6 rules I follow after 8 weeks of building with AI coding tools daily:

**1. Never merge code you can't debug.** If Claude wrote it and you can't trace the logic step by step, rewrite it or don't ship it. Speed now = pain later.

**2. AI is best at boilerplate, worst at architecture.** Let it write your CRUD endpoints, test fixtures, and config files. Never let it design your data model or choose your abstractions.

**6. The AI doesn't know your production environment.** It doesn't know your Fyers API rejects SL-M orders past a certain threshold. It doesn't know your database has a 5-second connection timeout. Always validate AI suggestions against your actual system's constraints.

AI coding tools are a 3x multiplier for people who already know what they're doing. They're a 0.5x multiplier for people who don't.
