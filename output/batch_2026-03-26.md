Here's what my setup looks like now:

```yaml
# openspec.yaml
workflows:
  post_generator:
    model: claude-sonnet-4-6
    system_prompt: !include prompts/linkedin_system.md
    temperature: 0.8
    max_tokens: 1000
    inputs:
      - trends: list
      - framework: string
    output_format: markdown

  trade_analyzer:
    model: llama3-local
    system_prompt: !include prompts/trade_analysis.md
    temperature: 0.2
    fallback: claude-sonnet-4-6
```

**What I like:**
- All prompts in one place with version control — I can `git diff` any prompt change

- No built-in prompt testing or evaluation framework
- Community is small — when you hit a bug, Stack Overflow has zero answers

**The honest verdict after 3 weeks:** My prompt management went from chaotic to boring. And boring infrastructure is the best kind.

If you're managing more than 3 LLM prompts, any structured system beats scattered files. OpenSpec is one option. LangChain has prompt templates too. The tool matters less than the discipline.

### Post 4: Edutainment: 1000 Hours -> 5 Minutes
IMAGE_QUERY: quantization INT8 Q4_K_M model size comparison chart neural network

I spent 3 weeks understanding LLM quantization. Here's what actually matters.

**The one-sentence version:** Quantization shrinks a model by using fewer bits per number. A 9B parameter model at bfloat16 takes ~18GB. At Q4_K_M (4-bit) it takes ~5GB. Same model, fits on your laptop.

ngrok just published "Quantization from the Ground Up" and it's the best explanation I've read. They tested everything on Qwen 3.5 9B. Here's what their benchmarks showed:

**The 3 methods that matter:**

1. **Symmetric (Q8_0):** Values squished around zero, one scale factor per block of 32-256 weights. File size just over half the original. Almost no quality loss.

2. **Asymmetric/Affine:** Allows uneven ranges with a zero-point offset. More accurate but stores both scale and offset per block. Better for models with uneven weight distributions.

3. **K-Quant (Q4_K_M):** The sweet spot. 4-bit with mixed precision — important layers get more bits, insensitive layers get fewer. This is what most GGUF models on Hugging Face use.

**The numbers that matter:**
- Q8_0 → half the size, virtually no quality loss
- Q4_K_M → 4x smaller, dequantized values off by ~18% on average — but model output quality barely changes for chat/summarization
- Q2_K → 8x smaller, but significant quality degradation
- Quantizing Qwen 3.5 9B took about **1 minute on a laptop**

**When NOT to quantize:** Math, precise code generation, or tasks where subtle wording matters. For summarization, classification, and chat? Q4_K_M is free performance.

Source: https://ngrok.com/blog/quantization

### Post 11: Listicle / Tips
IMAGE_QUERY: AI coding assistant rules developer workflow productivity tips

6 rules I follow after 2 months of daily AI-assisted coding:

**1. Never merge code you can't debug.** Mario Zechner (creator of the Pi agent framework) nailed it: agents have "no bottleneck, no pain signal." If Claude wrote it and you can't trace the logic, rewrite it. Speed now = pain later.

**2. AI is best at boilerplate, worst at architecture.** Let it write your CRUD endpoints, test fixtures, and config parsers. Never let it design your data model. The agent's decisions are always local — it "never sees the full codebase history" (Zechner's phrase).

**3. Prompt with constraints, not goals.** Bad: "Write a function to process orders." Good: "Function taking List[Order], filtering status=PENDING, returning total_value. No external deps. Under 20 lines."

**4. Read the diff, not the file.** When AI modifies existing code, `git diff` matters more than reading the full output. It shows exactly what changed and what might break.

**5. Check your Copilot privacy settings.** As of April 24, 2026, GitHub uses your interaction data for training unless you opt out. Three separate toggles. Go to github.com/settings/copilot/features and check all three.

**6. The AI doesn't know your production quirks.** It doesn't know Fyers API rejects SL-M orders when the trigger is "too far" from LTP. It doesn't know Apple auto-closes bug reports after 2 weeks. Always validate against your actual system.

AI coding tools are a 3x multiplier for people who already understand what they're building. They're a 0.5x multiplier for people who don't.
